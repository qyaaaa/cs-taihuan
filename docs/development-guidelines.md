# 项目开发规范

本文档记录 CS 汰换工作台的工程约定。新增功能、修 bug、补测试时优先遵循这里的规则，再参考具体模块已有写法。

## 1. 后端错误文案

- 面向接口返回、任务状态、参数校验的错误文案统一放在 `ErrorMessages`。
- 动态文案用静态方法生成，例如带 `snapshotId`、`remainingCount` 的提示。
- 业务层不要直接写 `throw new IllegalArgumentException("...")`，应引用 `ErrorMessages`。
- DTO 注解的 `message` 也应使用 `ErrorMessages` 中的 `public static final` 常量。

## 2. 复杂方法注释

- 复杂业务方法上方必须有简短注释，说明业务意图和关键约束。
- 代码注释统一使用中文；英文只作为必要的字段名、接口名、协议名、枚举值或第三方专有名词保留。
- 注释解释“为什么这样做”，不要逐行复述代码。
- 汰换概率、磨损公式、价格档匹配、BUFF 限流、目录同步队列等逻辑都属于必须注释的复杂逻辑。
- 简单 getter、setter、薄包装方法不用强行添加注释。

## 3. 汰换计算测试

- 核心计算优先补纯单元测试，不依赖 Spring、MySQL、BUFF 网络。
- 测试样例放在 `src/test/resources/fixtures/` 下，并使用脱敏数据。
- `TradeUpOptimizer` 回归测试至少覆盖：
  - 常规 10 合 1 概率和磨损公式
  - `covert -> gold` 5 合 1
  - StatTrak 与普通不可混放
  - StatTrak 金色产物过滤手套
  - 精细价格档和特殊价格因子
- 预期结果应落到 fixture 中，便于人工核对和后续回归。

## 4. 目录同步

- 目录同步默认由后端定时任务温和补齐，前端按钮是立即兜底触发。
- 定时任务和手动任务必须共用服务级互斥锁，避免同时请求 BUFF 详情接口。
- 同步逻辑必须尊重 `cache-fresh-millis`，近期已获取的 goods 不重复请求。
- BUFF 请求间隔不要低于配置保护值，避免触发限流。

## 5. 前端交互

- 长任务页面必须有 loading 状态，避免空态和旧数据闪烁。
- 库存、方案、目录同步、会话校验都应通过页面级或组件级 loading 明确反馈。
- UI 保持 BUFF 暗色工作台风格，主色以蓝紫为主，收益和价格点缀可使用金色。
- 图标资源放在 `frontend/public/`，入口 favicon 在 `frontend/index.html` 维护。

## 6. 数据安全

- `data/` 下的 Cookie、库存 JSON、临时输出不得提交到 Git。
- Cookie 只在本机保存，保存文件需要尽量限制为当前用户可读写。
- 文档和测试 fixture 不允许包含真实 Cookie、Steam ID、BUFF 账号、真实库存截图等敏感信息。

## 7. 多账号隔离

- Cookie、会话状态、库存快照、目录同步任务、方案生成结果必须按 `account_id` 隔离。
- `catalog_skin` 是全局市场目录，不加 `account_id`，不同账号共享同一份商品和价格数据。
- 新增账号接口时应保留旧接口兼容，旧接口内部走默认账号，避免前端灰度期间中断。
- 前端切换账号后必须刷新会话、库存和方案状态，不能继续展示上一个账号的数据。

## 8. 提交前检查

- 前端改动至少运行 `npm run build`。
- Java 改动至少运行 `mvn test`；如果本机没有 Maven，需要在交付说明里明确写出未运行原因。
- 提交前运行 `git diff --check`，避免空白字符和换行问题。
